Command Reference

Here's the thing about slash commands: they're workflows you'd spend 20 minutes doing manually, compressed into one line. Type /plan and watch three agents launch in parallel to research your codebase while you grab coffee. That's the point--automation that actually saves time.

Workflow Commands (5)

These are the big five: Brainstorm ideas, Plan your feature, Review your code, Work through the implementation, and Compound what you learned. Every professional developer does this cycle--these commands just make you faster at it.

/brainstorm

Got a vague idea that needs structure? This command runs a collaborative brainstorming session--exploring possibilities, challenging assumptions, and turning rough concepts into actionable directions.

Arguments

[idea, concept, or problem to brainstorm about]

/brainstorm How should we handle AI-powered content generation in our theme?
/brainstorm Plugin architecture for a custom booking system
/plan

You've got a feature request and a blank page. This command turns "we need custom blocks" into a structured plan that actually tells you what to build--researched, reviewed, and ready to execute.

Arguments

[feature description, bug report, or improvement idea]

Workflow

  1. Repository Research (Parallel) - Launch three agents simultaneously:
    • repo-research-analyst - Project patterns
    • best-practices-researcher - Industry standards
    • framework-docs-researcher - Framework documentation
  2. SpecFlow Analysis - Run spec-flow-analyzer for user flows
  3. Choose Detail Level:
    • MINIMAL - Simple bugs/small improvements
    • MORE - Standard features
    • A LOT - Major features with phases
  4. Write Plan - Save as plans/<issue_title>.md

This command does NOT write code. It only researches and creates the plan.

/plan Add custom Gutenberg blocks for the portfolio section
/plan Fix query performance on the archive page
/review

Specialized reviewers examine your PR in parallel--security, performance, architecture, WordPress PHP, hooks, Gutenberg blocks, and more. It's like code review by committee, except the committee finishes in two minutes instead of two days.

Arguments

[PR number, GitHub URL, branch name, or "latest"]

Workflow

  1. Setup - Detect review target, optionally use git-worktree for isolation
  2. Launch Parallel Review Agents:
    • wp-php-reviewer, wp-hooks-reviewer
    • security-sentinel, performance-oracle
    • architecture-strategist, data-integrity-guardian
    • pattern-recognition-specialist, call-chain-verifier
    • And more...
  3. Ultra-Thinking Analysis - Stakeholder perspectives, scenario exploration
  4. Simplification Review - Run code-simplicity-reviewer
  5. Synthesize Findings - Categorize by severity (P1/P2/P3)
  6. Create Todo Files - Using file-todos skill for all findings

P1 (Critical) findings BLOCK MERGE. Address these before merging.

/review 42
/review https://github.com/owner/repo/pull/42
/review feature-branch-name
/review latest
/work

Point this at a plan file and watch it execute--reading requirements, setting up environment, running tests, creating commits, opening PRs. It's the "just build the thing" button you wish you always had.

Arguments

[plan file, specification, or todo file path]

Phases

  1. Quick Start
    • Read plan & clarify requirements
    • Setup environment (live or worktree)
    • Create TodoWrite task list
  2. Execute
    • Task execution loop with progress tracking
    • Follow existing patterns
    • Test continuously
  3. Quality Check
    • Run test suite
    • Optional reviewer agents for complex changes
  4. Ship It
    • Create commit with conventional format
    • Create pull request
    • Notify with summary
/work plans/custom-blocks.md
/work todos/042-ready-p1-performance-issue.md
/compound

Just fixed a gnarly bug? This captures the solution before you forget it. Agents analyze what you did, why it worked, and how to prevent it next time. Each documented solution compounds your team's knowledge.

Arguments

[optional: brief context about the fix]

Workflow

  1. Preconditions - Verify problem is solved and verified working
  2. Launch parallel subagents:
    • Context Analyzer - Extract YAML frontmatter skeleton
    • Solution Extractor - Identify root cause and solution
    • Related Docs Finder - Find cross-references
    • Prevention Strategist - Develop prevention strategies
    • Category Classifier - Determine docs category
    • Documentation Writer - Create the file
  3. Create Documentation - File in docs/solutions/[category]/

Auto-Triggers

Phrases: "that worked", "it's fixed", "working now", "problem solved"

/compound
/compound WordPress REST API authentication fix

Utility Commands (16)

The supporting cast--commands that do one specific thing really well. Generate changelogs, resolve todos in parallel, triage findings, create new commands. The utilities you reach for daily.

/agent-native-audit

Audit your features for agent-native parity. Makes sure anything a user can do through the UI, an agent can also do through APIs or tools. 

/agent-native-audit
/changelog

Turn your git history into a changelog people actually want to read. Breaking changes at the top, fun facts at the bottom, everything organized by what matters to your users.

Arguments

[optional: daily|weekly, or time period in days]

Output Sections

  • Breaking Changes (top priority)
  • New Features
  • Bug Fixes
  • Other Improvements
  • Shoutouts
  • Fun Fact
/changelog daily
/changelog weekly
/changelog 7
/create-agent-skill

Need a new skill? This walks you through creating one that actually works--proper frontmatter, clear documentation, all the conventions baked in.

Arguments

[skill description or requirements]

/create-agent-skill WordPress block pattern library
/create-agent-skill Custom post type scaffolding
/deepen-plan

Already have a plan but it needs more detail? This command takes an existing plan and deepens it with additional research, edge cases, and implementation specifics.

Arguments

[plan file path]

Validate and prepare documentation for deployment. Checks links, formatting, and completeness before publishing. 

/deploy-docs

[optional: specific issue to fix]

/heal-skill parameter validation error

/report-bug

[optional: brief description of the bug]

Give it a GitHub issue number and it tries to actually reproduce the bug--reading the issue, analyzing code paths, iterating until it finds the root cause.

Arguments

[GitHub issue number]

/resolve_parallel

All those TODO comments scattered through your codebase? This finds them, builds a dependency graph, and spawns parallel agents to resolve them all at once.

Arguments

[optional: specific TODO pattern or file]

/resolve_parallel
/resolve_parallel authentication
/resolve_parallel src/auth/
/resolve_todo_parallel

Those todo files in your /todos directory? Point this at them and watch parallel agents knock them out--analyzing dependencies, executing in the right order, marking resolved as they finish.

Arguments

[optional: specific todo ID or pattern]

/resolve_todo_parallel
/resolve_todo_parallel 042
/resolve_todo_parallel p1
/slfg

Super quick-start variant of /lfg. Even faster to get coding. 

/slfg
/test-browser

Run browser-based tests to validate UI behavior and visual correctness. 

/test-browser
/triage

Got a pile of code review findings or security audit results? This turns them into actionable todos--one at a time, you decide: create the todo, skip it, or modify and re-present.

Arguments

[findings list or source type]

User Decisions

  • "yes" - Create/update todo file, change status to ready
  • "next" - Skip and delete from todos
  • "custom" - Modify and re-present
